Personal Study Journal Tips

Purpose

A Personal Study Guide is a space for you to document your learning throughout the course actively. It is not an assignment, and it is not graded. Its purpose is to help you think, practice, and connect ideas over time in ways that work for you.

You will keep your study guide inside your course GitHub repository, so it evolves alongside your code, projects, and labs. Think of this as a developer’s learning notebook, not a set of polished notes.

What This Is (and Is Not)

This is

  • A personal learning resource you build for yourself
  • A place to record understanding, confusion, examples, and patterns
  • Flexible in format and content
  • Something you can return to when preparing for exams, projects, or interviews

This is not

  • A required or graded deliverable
  • Something meant to be perfect, complete, or public-facing
  • A copy of the textbook, slides, or someone else’s notes

Where It Lives

  • Your study guide should live in your course GitHub repository.
  • A study-guide or journal folder in the repository
  • A NOTES.md or STUDY.md file, or other documents or recordings
  • Multiple markdown files organized by topic or week
  • You may use any structure that makes sense to you.

What You Can Include

Your study guide can take many forms. Writing is common, but not required. You may mix formats freely.

Written Notes

  • Explanations in your own words
  • Short summaries of concepts
  • “Here is what finally made this click for me”
  • Lists of common mistakes or pitfalls

Code and Examples

  • Small, focused code examples
  • Variations of examples discussed in class
  • “What happens if I change this?” experiments
  • Annotated snippets explaining why something works

Diagrams and Images

  • Hand-drawn diagrams (scanned or photographed)
  • Architecture sketches
  • Flowcharts or schema diagrams
  • Screenshots with notes

Recordings

  • Short audio or video explanations to yourself
  • Walkthroughs of a concept or solution
  • Screen recordings showing code behavior

Questions and Reflections

  • Questions you still have
  • Things you struggled with and how you resolved them
  • Connections between topics
  • Notes on feedback you received

Course-Specific Suggestions

You are encouraged to tailor your study guide to the course.

Java Programming

You might include:

  • Notes with page numbers referring to the books
  • Class and object diagrams
  • Examples showing features
  • Notes on the subtleties of Java semantics
  • Explanations of common errors and exceptions
  • Comparisons between similar constructs
  • Short programs that isolate one concept

Web Page Programming

You might include:

  • Notes from reading the material
  • HTML structure patterns, tags, features.
  • CSS layout diagrams and experiments
  • JavaScript behavior walkthroughs
  • DOM manipulation examples
  • Notes on browser behavior or debugging techniques

Advanced Database Course

You might include:

  • Notes with page numbers referring to the books
  • Schema designs and ER diagrams
  • Normalization examples and tradeoffs
  • Query patterns with explanations
  • Execution plan observations
  • Reflections on performance, indexing, or design decisions

How Detailed Should It Be?

As detailed as you need it. Some students write a few bullet points per topic. Others build extensive explanations with examples. Both are valid. The goal is usefulness, not volume. If you can come back later and say, “This helps me understand or remember,” then it is doing its job.

A Final Note

Because this study guide is optional and ungraded, you are free to experiment. If something does not work for you, change it. If a format helps you learn better, lean into it. Professional developers constantly build personal notes, scratch files, and learning artifacts. This is practice for that habit.
